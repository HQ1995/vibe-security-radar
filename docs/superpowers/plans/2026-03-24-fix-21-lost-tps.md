**Goal:** Fix the three code-level root causes behind TPs lost in the screening pipeline regression test.

## Root Causes (post-audit)

### RC1: Concurrent blame on same repo (15 CVEs)

Multiple batch workers run `git blame` on the same blobless clone simultaneously. Blobless clones fetch blobs on demand — concurrent fetches cause silent failures (empty blame output, 0 BICs). Affects repos with many CVEs (openclaw: 244 empty out of 316).

**Evidence:** `git blame` works perfectly in isolation (single-CVE rerun gets 20 BICs). Batch with 64 workers produces 0 BICs for the same CVE. Phase B completes in 0.05s (vs normal 13s).

**Affected:** 15 CVEs (openclaw + langbot repos)

### RC2: Screening rejects CVEs with existing confirmed BICs (2 CVEs)

`screen_cve()` evaluates AI atomic commits and their file relevance to the vulnerability. But it can miss cases where a different AI commit (not the one it evaluated) was previously deep-verify CONFIRMED. The screening prompt doesn't see deep verification history.

**Affected:** CVE-2026-32322 (rs-soroban-sdk), GHSA-qr2g-p6q7-w82m (x402)

### RC3: `_cap_bics` AI preservation is dead code (1 CVE)

`_cap_bics` (pipeline.py:1837-1870) has logic to preserve AI-signaled BICs when capping to 20. But `_apply_post_blame_filters` (which calls `_cap_bics`) runs at line 3478, BEFORE signal detection at line 3487. At cap time, `bic.commit.ai_signals` is always empty, so the preservation logic never fires. Non-deterministic blame can then displace the AI-signaled BIC with a higher-confidence non-AI BIC.

**Audit confirmed:** GHSA-gc24-px2r-5qmf (bambuddy) — Claude Code wrote an unauthenticated debug endpoint, but the BIC was dropped by the cap.

### Not bugs (audit cleared)

- **CVE-2025-53108** (homebox): Old pipeline was a **false positive**. Copilot sub-commit only changed a log message, not the authorization flaw. New pipeline correctly says UNLIKELY.
- **CVE-2026-27900** (terraform-provider-linode): Old pipeline was a **false positive**. Copilot trailer on squash merge, but Copilot sub-commits touched firewall/VPC files, not the log-sensitive-data files. New pipeline correctly excludes.

## Task 1: Blame clone pool — isolate concurrent blame via separate clones

**Files:**
- Modify: `cve-analyzer/src/cve_analyzer/git_ops.py` — add `BlameClonePool` and `acquire_blame_clone`/`release_blame_clone`
- Modify: `cve-analyzer/src/cve_analyzer/pipeline.py` — use pool clone for blame in `_blame_one_fix_commit`
- Test: `cve-analyzer/tests/test_blame_clone_pool.py`

**Context:** `git blame` on blobless clones triggers on-demand blob fetches. Concurrent fetches on the **same `.git` directory** corrupt each other (pack file race). A per-repo mutex would serialize blame but kill throughput for repos with many CVEs (openclaw: 316 CVEs, ~13s each = 68min serialized).

Instead, maintain a small **pool of independent clones** per repo. Each pool clone has its own `.git` directory — fully isolated. Workers acquire a clone from the pool, blame it, then release. If all clones are busy, the worker blocks until one is free.

**Design:**
- Pool max size: 4 per repo (enough for typical concurrency, minimal disk)
- Clone 0: the existing primary clone (from `clone_repo()`)
- Clones 1-3: created on demand as `<primary>.pool-1`, `<primary>.pool-2`, etc.
- Each pool clone is a fresh `git clone --filter=blob:none` (not `--reference` — full isolation, no shared state)
- Blobless clones are ~few MB (tree+commit objects only), so 4× is negligible
- Pool is per-repo-path, managed by a `BlameClonePool` class in `git_ops.py`

**Interface:**
```python
# In git_ops.py
class BlameClonePool:
    """Pool of isolated blobless clones for concurrent blame operations."""
    def __init__(self, max_pool_size: int = 4): ...
    def acquire(self, primary_path: Path, repo_url: str) -> Path: ...
    def release(self, clone_path: Path) -> None: ...

_blame_pool = BlameClonePool()
```

**Usage in pipeline.py** — `_blame_one_fix_commit` (line ~2340) is where per-fix-commit blame happens. Change from:
```python

# ... blame_lines(local_path, ...) ...
```
To:
```python

    # ... blame_lines(pool_clone, ...) ...
finally:
    git_ops._blame_pool.release(pool_clone)
```

Or use a context manager:
```python
with git_ops.blame_clone(fc.repo_url, shallow_since=shallow_since) as clone_path:
    # ... blame_lines(clone_path, ...) ...
```

Add to `git_ops.py`:

```python
class BlameClonePool:
    """Pool of isolated blobless clones for safe concurrent blame.

    Blobless clones fetch blobs on demand during blame. Concurrent fetches
    on the same .git directory corrupt pack files. This pool maintains
    separate clone directories per repo so each blame worker gets its own
    isolated .git.

    Pool clones are named <primary>.pool-1, .pool-2, etc. Created on demand,
    up to max_pool_size per repo. Workers block if all clones are busy.
    """

    def __init__(self, max_pool_size: int = 4):
        self._max = max_pool_size
        self._lock = threading.Lock()
        # repo_path → list of (clone_path, in_use: bool)
        self._pools: dict[str, list[tuple[Path, bool]]] = {}

    def acquire(self, primary_path: Path, repo_url: str,
                shallow_since: str | None = DEFAULT_SHALLOW_SINCE) -> Path:
        """Get an available clone for blame. Blocks if pool is full."""
        key = str(primary_path)
        while True:
            with self._lock:
                pool = self._pools.setdefault(key, [(primary_path, False)])
                # Find an available clone
                for i, (path, in_use) in enumerate(pool):
                    if not in_use:
                        pool[i] = (path, True)
                        return path
                # All busy — create new clone if under limit
                if len(pool) < self._max:
                    idx = len(pool)
                    new_path = Path(f"{primary_path}.pool-{idx}")
                    pool.append((new_path, True))
                    # Release lock before cloning (slow I/O)
                    # Mark as in_use so no one else tries the same slot
                    break
                else:
                    # Pool full — need to wait
                    pass  # fall through to sleep+retry

            # Either clone new or wait
            if 'new_path' in dir():
                # Clone outside the lock
                if not new_path.exists():
                    _create_pool_clone(primary_path, new_path, repo_url, shallow_since)
                return new_path

            # Pool full, wait and retry
            import time
            time.sleep(0.1)

    def release(self, clone_path: Path) -> None:
        """Return a clone to the pool."""
        with self._lock:
            for pool in self._pools.values():
                for i, (path, in_use) in enumerate(pool):
                    if path == clone_path:
                        pool[i] = (path, False)
                        return
```

Note: The above is a sketch — the implementer should use a proper `threading.Condition` instead of sleep+retry for the "pool full" case. Also needs `_create_pool_clone()` helper.

Add a context manager convenience:

```python
@contextlib.contextmanager
def blame_clone(url: str, shallow_since: str | None = DEFAULT_SHALLOW_SINCE):
    """Context manager: acquire a pool clone for blame, release on exit."""
    primary = clone_repo(url, shallow_since=shallow_since)
    if primary is None:
        yield None
        return
    clone = _blame_pool.acquire(primary, url, shallow_since)
    try:
        yield clone
    finally:
        _blame_pool.release(clone)
```

**Files:**

Two options:
- **Option A**: Move `_apply_post_blame_filters` to after Phase B.2 (signal detection). Risk: other filters in the function may depend on running before signals.
- **Option B**: Change `_keep_top_bics` to use `b.all_ai_signals()` instead of `b.commit.ai_signals`. Simpler, no ordering change.

Option B is safer — `all_ai_signals()` checks decomposed commits which ARE populated by Phase B.1 (decomposition runs at line 3482, before cap at 3478... wait, let me check).

Actually the ordering is: line 3476 `_apply_post_blame_filters` → line 3482 decompose → line 3487 signals. So decomposed commits are ALSO empty at cap time. Option B won't help either.

**Correct fix**: Split `_apply_post_blame_filters` — run the non-cap filters (fix-SHA removal, context_blame demotion) before decomposition, but run `_cap_bics` AFTER signal detection. Or simpler: just move the `_cap_bics` call to after Phase B.2.
