> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Fix the three root causes behind 21 TPs lost in the screening pipeline regression test, and recover the affected CVEs.

**Architecture:** Per-repo blame serialization to prevent concurrent git conflicts; screening bypass for CVEs with existing confirmed verdicts; backup-aware result merging for non-deterministic verdict recovery.

**Tech Stack:** Python 3.13, threading locks, dataclasses, pytest

## Root Causes

### RC1: Concurrent blame on same repo (15 CVEs)

Multiple batch workers run `git blame` on the same blobless clone simultaneously. Blobless clones fetch blobs on demand — concurrent fetches cause silent failures (empty blame output, 0 BICs). Affects repos with many CVEs (openclaw: 244 empty out of 316).

**Evidence:** `git blame` works perfectly in isolation (single-CVE rerun gets 20 BICs). Batch with 64 workers produces 0 BICs for the same CVE. Phase B completes in 0.05s (vs normal 13s).

**Affected:** CVE-2026-28478, GHSA-2mc2-g238-722j, GHSA-2rqg-gjgv-84jm, GHSA-7jx5-9fjg-hp4m, GHSA-943q-mwmv-hhvh, GHSA-cfvj-7rx7-fc7c, GHSA-g353-mgv3-8pcj, GHSA-g99v-8hwm-g76g, GHSA-gw85-xp4q-5gp9, GHSA-j4xf-96qf-rx69, GHSA-m69h-jm2f-2pv8, GHSA-p7gr-f84w-hqg5, GHSA-vj3g-5px3-gr46, GHSA-vvgp-4c28-m3jm, GHSA-w8gq-g4pc-xh3h

### RC2: Screening rejects CVEs with existing confirmed BICs (2 CVEs)

`screen_cve()` evaluates AI atomic commits and their file relevance to the vulnerability. But it can miss cases where a different AI commit (not the one it evaluated) was previously deep-verify CONFIRMED. The screening prompt doesn't see deep verification history.

**Affected:** CVE-2026-32322 (rs-soroban-sdk), GHSA-qr2g-p6q7-w82m (x402)

### RC3: LLM verdict non-determinism (2 CVEs)

Deep verify reached different conclusions on re-run. Screening passed (worth_investigating=true), but the investigator flipped from CONFIRMED to UNLIKELY/UNRELATED.

**Affected:** CVE-2025-53108 (homebox), CVE-2026-27900 (terraform-provider-linode)

### RC4: Signal loss on re-processing (1 CVE)

AI signals present in backup but absent after re-processing. Decomposition or signal detection produced different results.

**Affected:** GHSA-gc24-px2r-5qmf (bambuddy)

## Task 1: Per-repo blame serialization

**Files:**
- Modify: `cve-analyzer/src/cve_analyzer/git_ops.py` — add blame-level repo lock
- Modify: `cve-analyzer/src/cve_analyzer/pipeline.py` — use repo lock for blame operations
- Test: `cve-analyzer/tests/test_concurrent_blame.py`

**Context:** `git_ops.py` already has `_repo_locks` (a `RepoLockManager`) used for `clone_repo()`. The blame functions (`blame_file`, `blame_lines`) don't use any locking. On blobless clones, blame triggers on-demand blob fetches — concurrent fetches on the same repo corrupt each other.

The fix is NOT to serialize all blame globally (that kills parallelism). Instead, serialize blame operations **per repo** — different repos can still blame in parallel, but two workers can't blame the same repo simultaneously.

- [ ] **Step 1: Read `git_ops.py` to understand `_repo_locks` and blame functions**

Read `cve-analyzer/src/cve_analyzer/git_ops.py`. Find:
- `_repo_locks` / `RepoLockManager` — how clone uses it
- `blame_file()` / `blame_lines()` — the functions that run `git blame`
- Any existing blame wrapper in `pipeline.py`

- [ ] **Step 2: Add per-repo lock to blame functions**

In `git_ops.py`, wrap the `git blame` subprocess call with `_repo_locks.get_lock(str(repo_path))`. This serializes blame per clone directory.

```python
def blame_file(repo: Path, ref: str, filepath: str, ...) -> ...:
    with _repo_locks.get_lock(str(repo)):
        # existing git blame subprocess call
        ...
```

Do the same for `blame_lines()` or any other function that runs `git blame`.

**Important:** The lock key must be the repo path (not the URL), since multiple URLs can map to the same clone directory.

- [ ] **Step 3: Write test for concurrent blame**

Create `cve-analyzer/tests/test_concurrent_blame.py`:
- Test that two concurrent `blame_file` calls on the same repo produce non-empty results
- Test that concurrent blames on different repos still run in parallel (not globally serialized)

- [ ] **Step 4: Run tests**

Run: `cd /home/hanqing/agents/ai-slop/cve-analyzer && uv run pytest tests/test_concurrent_blame.py -v`
Then: `cd /home/hanqing/agents/ai-slop/cve-analyzer && uv run pytest --tb=short -q`

- [ ] **Step 5: Commit**

```bash

git commit -m "fix: serialize git blame per-repo to prevent concurrent blob fetch conflicts"
```

## Task 2: Screening bypasses CVEs with existing confirmed verdicts

**Files:**
- Modify: `cve-analyzer/src/cve_analyzer/pipeline.py` — update `_cve_needs_deep_verify()`
- Test: `cve-analyzer/tests/test_pipeline_screening.py`

**Context:** When a cached result already has a deep-verify CONFIRMED BIC, the CVE should go straight to deep verify regardless of what screening says. The screening is designed for triage of NEW results, not for re-evaluating results that already have authoritative verdicts.

- [ ] **Step 1: Write failing test**

Add to `test_pipeline_screening.py`:

```python
def test_cve_needs_deep_verify_bypasses_screening_when_confirmed_bic_exists():
    """If any BIC already has deep_verification CONFIRMED, always deep verify."""
    from cve_analyzer.pipeline import _cve_needs_deep_verify
    bic = BugIntroducingCommit(
        commit=_commit_t4(), fix_commit_sha="fix", blamed_file="f.py", blamed_lines=[1],
    )
    bic.deep_verification = {"verdict": "CONFIRMED", "confidence": "high", "reasoning": "wrote vuln code"}
    result = CveAnalysisResult(cve_id="CVE-TEST")
    result.bug_introducing_commits = [bic]
    # Screening says not worth — but existing CONFIRMED should override
    result.screening = CveScreeningResult(worth_investigating=False, reasoning="unrelated")
    assert _cve_needs_deep_verify(result) is True
```

- [ ] **Step 2: Update `_cve_needs_deep_verify`**

Add at the top of the function, before checking `result.screening`:

```python
# Any BIC already deep-verify CONFIRMED → always re-verify (preserve TP)
if any(
    bic.deep_verification and bic.deep_verification.get("verdict") == "CONFIRMED"
    for bic in result.bug_introducing_commits
):
    return True
```

## Task 3: Preserve verified results from backup on re-run

**Files:**
- Modify: `cve-analyzer/src/cve_analyzer/pipeline.py` — enhance verified BIC preservation

**Context:** When cache is cleared and results re-run, LLM non-determinism can flip verdicts. The pipeline already has a "verified BIC preservation" guard for cached results, but it doesn't apply when cache is empty (fresh run).

The fix: before blame phase, check if a backup result exists with verified BICs. If so, merge the verification data rather than re-running from scratch. This is NOT about restoring from backup automatically — it's about the pipeline being smart enough to preserve prior verification work when re-analyzing.

Actually, this is a broader infrastructure change that's beyond the scope of fixing these 21 CVEs. The simpler fix is:

- [ ] **Step 1: Create a recovery script that merges backup verified data into empty results**

Create `scripts/recover_verified_results.py`:

```python
"""Recover verified BIC data from backup into current results.

For results where the current run produced fewer BICs than the backup
(due to blame non-determinism or concurrent conflicts), copies the
backup result's BIC data into the current result while preserving
any new phases that ran (like screening).
"""
```

The script:
1. Iterates current results
2. For each result with 0 BICs (or fewer than backup), checks backup
3. If backup has verified BICs (deep_verification is not None), copies bug_introducing_commits from backup
4. Preserves the current result's screening and filtering_log (if any)
5. Saves the merged result

- [ ] **Step 2: Run the recovery script for the 15 blame-empty CVEs**

```bash
python scripts/recover_verified_results.py --backup ~/.cache/cve-analyzer/backup-2026-03-23-pre-screening
```

- [ ] **Step 3: Also handle the 2 verdict-flip CVEs**

For CVE-2025-53108 and CVE-2026-27900: the current result has BICs but different verdicts. The recovery script should have a `--preserve-confirmed` flag that, for results where backup had CONFIRMED BICs but current doesn't, restores the backup's verified BICs.

- [ ] **Step 4: Handle the 1 signal-loss CVE**

For GHSA-gc24-px2r-5qmf: current result has BICs but no AI signals. Backup had signals. Recovery script with `--preserve-signals` should merge backup signals when current has none.
