# False Negative CVE Audit

Investigate CVEs where the pipeline found **no AI signals** to check if AI involvement was missed. This catches blind spots in pattern matching, squash-merge signal loss, and unknown AI tools.

**Arguments**: `$ARGUMENTS` (a CVE ID, or empty to auto-pick from FN queue)

## Critical: Bug Introduction vs Fix Authorship

**This audit investigates whether AI wrote the code that INTRODUCED the vulnerability.** Even when you discover hidden AI signals on a commit, verify that the AI-authored code is the *bug-introducing* code, not the *fix* code. Finding an AI co-author on a security patch does NOT mean AI introduced the bug.

## Why This Matters

The pipeline detects AI signals via co-author trailers, author emails, and commit message patterns. But it can miss:
- New AI tools not in the pattern list
- Signals hidden inside squash merges where sub-commits aren't checked
- AI involvement via PR review bots (not in commit metadata)
- Variant co-author formats the regex doesn't catch

Only by independently investigating "clean" CVEs can we measure and reduce the false negative rate.

## Phase 0: Select Target

If a CVE ID was provided, use it. Otherwise, use the FN audit queue:

```bash
python3 scripts/audit_queue.py --fn
```

This scores CVEs without AI signals by false-negative likelihood (author overlap with known AI users, squash-heavy repos, etc.) and recommends the highest-priority target.

## Phase 1: Load Cached Result

Load `~/.cache/cve-analyzer/results/<CVE-ID>.json`. Extract and print:

- `cve_id`, `description`, `severity`
- `fix_commits` — sha, repo_url, source
- `bug_introducing_commits` — sha, author, blamed_file, blame_strategy
- Confirm: **no `ai_signals`** on any BIC (if signals exist, this is an FP audit — use `/audit` instead)

## Phase 2: Understand the Vulnerability

Follow `/audit` Phase 2 exactly:
- 2a. Read the fix diff (`git show <fix_sha>`)
- 2b. Research the CVE (web search)
- 2c. Extract vulnerable pattern vs secure pattern

## Phase 3: Independent Blame

Follow `/audit` Phase 3 exactly:
- Blame the fix diff lines
- Compare with pipeline's BICs

## Phase 4: AI Signal Deep Investigation

This is the core of FN audit — go deeper than the pipeline's pattern matching.

### 4a. Full commit metadata

For each BIC:

```bash
git show --format=fuller <bic_sha>    # all headers + trailers
git log --format="%B" <bic_sha> -1    # full body including trailers
```

Look for:
- Co-author trailers with **variant formats** the pipeline might miss (e.g., `Co-Authored-By:` vs `Co-authored-by:`, tool name variations, non-standard email formats)
- Author/committer email from AI services
- Commit message style: overly structured, bullet-pointed, or template-like descriptions

### 4b. PR investigation

If the BIC has a PR (find via commit message `(#NNNN)` or `gh pr list --search <sha>`):

```bash
gh pr view <number> --repo <owner/repo> --json author,labels,comments,body
```

Check for:
- PR body mentioning AI tools ("generated by", "with help from", "suggested by Copilot")
- Bot review comments (Copilot, CodeRabbit, Sourcery, etc.)
- Labels like `ai-generated`, `copilot`, `bot`
- PR author's profile (is it a bot account?)

### 4c. Squash merge decomposition

If the BIC is a squash merge (committer is `noreply@github.com`):

```bash
# Find the PR
gh pr view <number> --repo <owner/repo> --json commits

# Check each sub-commit for AI signals
git show --format=fuller <sub_commit_sha>
```

The pipeline might have missed signals on individual sub-commits that got squashed away.

### 4d. Author context

Check if this author uses AI tools in other commits in the same repo:

```bash
git log --author="<author_email>" --format="%H %s" -20
# Then check a few for co-author trailers
git show --format=fuller <recent_commit> | grep -i "co-authored"
```

### 4e. Classify finding

After investigation, classify:
- **AI_FOUND** — discovered AI involvement the pipeline missed
- **NO_AI** — thorough investigation confirms no AI involvement
- **INCONCLUSIVE** — suspicious but no definitive evidence

## Phase 5: Causality Analysis (if AI found)

Only if Phase 4 result is `AI_FOUND`:

Follow `/audit` Phase 5 exactly — determine if the AI-authored code actually introduced the vulnerability.

Write verdict: `CONFIRMED` / `UNLIKELY` / `UNRELATED` with confidence.

If Phase 4 result is `NO_AI` or `INCONCLUSIVE`, skip this phase.

## Phase 6: Pipeline Diagnosis (if AI found)

If AI involvement was found, diagnose **why the pipeline missed it**:

| Miss Category | Description | Fix Location |
|--------------|-------------|--------------|
| `missing_pattern` | AI tool not in detection patterns | `repo_analyzer.py` signal patterns |
| `format_variant` | Co-author trailer format not matched | `repo_analyzer.py` regex |
| `squash_loss` | Signal on sub-commit lost in squash | `pr_enrichment.py` squash handling |
| `pr_only_signal` | Signal in PR metadata, not commit | New: PR metadata scanning |
| `bot_review` | AI involvement via PR review, not authorship | New: review comment scanning |

Provide specific details: what pattern/regex would catch this, which file needs the change.

## Phase 8: Pattern Analysis (every 10 FN findings)

After saving, count FN findings. If ≥10:

```python
python3 -c "
import json; from pathlib import Path; from collections import Counter
findings = [f for f in json.loads((Path.home() / '.cache/cve-analyzer/audit/findings.json').read_text()) if f.get('audit_type') == 'fn_detection']
print(f'FN audits: {len(findings)}')
found = sum(1 for f in findings if f.get('ai_found'))
print(f'AI actually found: {found}/{len(findings)} ({100*found/max(len(findings),1):.0f}%)')
miss_types = Counter(f.get('missed_signal_type') for f in findings if f.get('ai_found'))
print(f'Miss categories: {dict(miss_types)}')
no_ai = sum(1 for f in findings if not f.get('ai_found'))
print(f'Confirmed no AI: {no_ai} (pipeline was correct)')
"
```

Report:
1. FN detection rate (what % of "clean" CVEs actually had hidden AI involvement)
2. Most common miss category → prioritize pipeline fixes
3. Specific pattern/regex additions needed
